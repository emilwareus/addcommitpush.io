SlopCodeBench (Orlanski et al., 2024) evaluates iterative
specification updates and measures verbosity and structural erosion, showing that coding-agent-generated code

engine using autonomous coding agents (Wilson Lin, 2026)

autonomously build a C compiler (Carlini, 2026) produced

lacked human-like abstraction. These observations suggest
that while current methods excel at producing functional
patches, they struggle to produce code that remains comprehensible and extensible over time.

Large language model (LLM) coding agents can
generate working code, but their solutions often
accumulate complexity, duplication, and architectural debt. Human developers address such
issues through refactoring: behavior-preserving
program transformations that improve structure
and maintainability. In this paper, we investigate
if LLM agents (i) can execute refactorings reliably and (ii) identify the refactorings that human
developers actually chose in real codebases. We
present C ODE TASTE, a benchmark of refactoring
tasks mined from large-scale multi-file changes
in open-source repositories. To score solutions,
we combine repository test suites with custom
static checks that verify removal of undesired
patterns and introduction of desired patterns using dataflow reasoning.
Our experimental results indicate a clear gap
across frontier models: agents perform well
when refactorings are specified in detail, but often fail to discover the human refactoring choices
when only presented with a focus area for improvement. A propose-then-implement decomposition improves alignment, and selecting the
best-aligned proposal before implementation can
yield further gains. C ODE TASTE provides an
evaluation target and a potential preference signal for aligning coding agents with human refactoring decisions in realistic codebases.

Lack of challenging refactoring benchmarks Refactoring is the software engineering practice of applying
behavior-preserving transformations that improve structure and maintainability. OpenAI reports that they train
and evaluate their models on large-scale code refactoring
tasks, without disclosing their internal evaluation frameworks (OpenAI, 2025a). The academic community has
also contributed benchmarks for refactoring: RefactorBench provides few-file tasks with syntactic tests (Gautam
et al., 2025), and SWE-Refactor collects 1,099 refactoring
of a fixed set of method-level refactorings from 18 Java
repositories (Xu et al., 2026).

Our work In this work, we investigate this challenge
extensively and use precise code-level measurements to
assess whether state-of-the-art coding agents can autonomously discover and implement relevant refactorings
in real-world code bases. We introduce an alignment score
that measures how many occurrences of undesired coding
patterns are converted to desired versions, counting only
functionally correct implementations. Our results indicate
a clear gap: while frontier models score highly on implementing complex refactorings that are specified in detail, achieving up to 70% alignment in our evaluation, they
fail to mitigate undesired coding patterns correctly when
no concrete refactoring is specified, achieving overall less
than 8% alignment. We introduce C ODE TASTE, a benchmark designed to measure this gap. We publicly release its
leaderboard, our code and the dataset1

We observe that frontier models consistently outperform
lower capability models on our benchmark, with for example GPT-5 C ODEX achieving 7.7% alignment in the open
track direct setting compared to 2.3% for Q WEN 3. This
suggests that human refactoring preferences are already
partially captured by general model training. Still, our case
studies, we observe that current agents often take shortcuts
and tend towards apply minimal, insufficient changes, unless explicitly instructed to plan out a refactoring.

2. Methodology
In Figure 1, we provide a summary of the C ODE TASTE
pipeline. Based on a dataset of historic code changes, we
(i) mine large, multi-file refactorings performed by human
developers, (ii) translate each refactoring into task descriptions for our two tracks, (iii) build a reproducible execution environment that can run each repository’s unit tests,
and (iv) generate static analysis rules that capture desired
and undesired patterns of the transformation. We then run
agents to produce a candidate patch and evaluate it using
both the repository test suite and our static rules.

A scalable pipeline for discovering refactoring instances. We mine popular GitHub repositories for 100
large multi-file refactorings performed by human developers. For each task, we generate an executable environment and static analysis rules that match the changed code
patterns. The latter support semantic dataflow reasoning
within files, which provides more flexibility than syntactic
matching, and is automatically verifiable. After the model
applies a refactoring, we measure performance, ensuring
that (i) the repository test suite passes and (ii) the static
analysis rules confirm code pattern transformation.

During evaluation, we task coding agents to reimplement
the original human refactoring. These agents can interact
with the task description, the codebase, and the test suite,
and the execution environment, producing a final predicted
patch. Finally, we evaluate this patch using (i) the test suite
to validate whether the functionality has been preserved
and (ii) the static analysis rules to validate how consistently
the refactoring was implemented.

We first introduce the notation to describe codebases R,
their test suites T , and changes to these codebases in the
form of patches X. Following the notation of Mündler et al.
(2024), we denote as R ◦ X a codebase R after applying
patch X. In this case, the human-authored base commit is
R. The developer written golden patch is denoted X ∗ , and
the corresponding golden state or golden commit is R◦X ∗ .

Our current pipelines and filters are designed to generate
a benchmark of relatively small size, due in large part to
the high cost of performing a refactoring, often exceeding
10 USD per task. Our collection using can be scaled up
trivially by leveraging additional base databases and modifying its filter criteria.

On any codebase R, a set of tests T can be used to check
the correctness of the applied patch. Executing a test suite
T of repository R runs a total number of tests |T | and returns execR (T ) = (PR , FR ) a count of passing tests PR
and failing tests FR . If execution of the test suite fails unexpectedly, e.g., due to a crash, we set (PR , FR ) = (0, 0).

3.2. Task generation
For each task instance, we generate a structured, highly
detailed task description modeled after a human-written
GitHub issue using an LLM. The description is synthesized from the commit message, associated Pull Requests,
including titles, bodies, and linked issues, and the code of
the golden patch X ∗ .

Let LR be the set of lines in R, where each l ∈ LR is a

and line content, in this order. A patch X induces a partial

X (l ≡X l′ ) iff MX (l) = l′ . Lines deleted by X are

with a small LLM and keep the highest-ranked 160 candidates. We run these three steps four times, once for each of

whether the addition is valid and distinct, and (iii) a submission command to stop the interaction. We detail the

3.5. C ODE TASTE
We apply the described pipeline, using S ONNET 4.5 to synthesize task descriptions, set up build environments and
generate static rules. We thus obtain C ODE TASTE, our
main benchmark, consisting of 100 instances distributed
across 87 repositories and 6 programming languages. The
detailed distribution across languages is shown in Figure 5.
The benchmark comprises complex and large refactorings,
as shown in Table 1, with instances on average requiring
edits to 91.52 files and 2605.39 lines of code. The most
complex tasks require up to 18821 line changes across up
to 290 files. To validate correctness, the benchmark runs an
average of 1638.53 tests and validates an average of 29.66
additive and 63.41 reductive rules per instance.

3.4. Rule Generation
To capture the semantic intent of a code transformation
without overfitting to specific concrete changes, we frame
rule generation as a search task performed by an LLMbased agent. The agent is only provided the task description generated in the previous step, to avoid overfitting to
the actual code changes in X ∗ . For each task instance, the
agent is then tasked to generate a set of static analysis rules
Γ = Γ+ ∪ Γ− using OpenGrep syntax. These rules use patterns over the AST with optional intra-file dataflow (e.g.,
symbolic propagation and taint mode) to describe the transformation abstractly.

Even in the instructed track, where agents are provided
with a detailed blueprint specifying the exact refactoring transformations required, notable performance gaps
emerge among frontier systems. Unless otherwise specified, all metrics reported in this section and in §4.3 concern
averages across instances.
SOTA agents have major differences As can be seen
in Figure 7a, GPT-5.2 significantly outperforms all other
agents on alignment A, achieving an average A of 69.6%.
In contrast, S ONNET 4.5 and GPT-5.1 M achieve only
32.4% and 34.6%, respectively. The open-source Q WEN 3
lags substantially behind at 11.8%.
Notably, all of GPT-5.2, GPT-5.1 M and S ONNET 4.5
have high I FR of 89.3%, 72.2% and 69.2%. However, the
A of the latter two is significantly lower than for GPT5.2, since their PASS is roughly 30 percentage-points below GPT-5.2’s 76.0% at 47.0% and 43.0%, respectively.
Thus, while all frontier agents are able to follow detailed
and large scale changes, there are large gaps in maintaining
code correctness. In our case studies, we find that GPT-5.2
achieves higher PASS rates by applying targeted, individual
patches, whereas S ONNET 4.5 runs insensitive search and
replace commands. While one might suspect that propensity to validate outputs with the test suite might have a
strong effect, GPT-5.2 actually runs tests less frequently

4.4. Failure Modes in Open Track

Achieving high I FR on the open track is significantly
harder. This is because agents are only provided with a
high-level focus area and need to recover human architectural choices. The maximal achieved alignment score A
over all models is 7.7% in Direct Inference mode, as can
be seen in Figure 7b.

To answer what the open track actually tests, we investigated a sample of solutions predicted by GPT-5.2 and
S ONNET 4.5 and compared the golden solution with their
solutions. Some observed issues concern (a) fixating on
salient, obvious subjects, e.g., a typo in the name of a class,
and completely ignoring the broader focus area, (b) resorting to lazy, hacky workarounds, e.g., creating a compatibility shim to bypass updating hundreds of import statements, (c) relying on destructive, repository-wide replaceall string commands, leaving the codebase in a broken and
inconsistent state, and (d) working on entirely different aspects of the system than required. While we cannot predict
what a realistic upper bound for human alignment is on a
refactoring, we observe that even the best current models
have major issues choosing appropriate refactorings. We
include examples of LLM-assisted manual analysis on such
cases in Appendix C.

Models need planning for large refactorings When the
models first propose a plan before implementing it (Plan
Mode), there is a significant increase in A. For GPT-5.2,
the A nearly doubles from 7.7% to 14.1%. On average,
the scores increase by a total of almost 3%, corresponding to a more than 50% relative increase. This suggests
that explicit reasoning before editing helps in recovering
large-scale human refactoring intents. Case studies (Appendix C.1) reveal that planning tends to yield repositorywide transformations that reflect high-level architectural
thinking. The corresponding patches are approximately 2
to 3 times larger (measured by |LX̂ |) and achieve up to
a 72% relative increase in I FR. The broader scope can
come at the cost of up to 24% lower PASS rates (Direct
vs. Plan). Q WEN 3 exhibits the most significant decline,
whereas GPT-5.2 sees no decline. Details across all models and modes are presented in Appendix Table 4.

5. Related Work
Prior benchmarks have measured more basic refactorings
conducted by LLMs or coding agents, which we detail in
this section.
Single-File Refactoring Benchmarks A number of
benchmarks measure changes only on single-file edits.
RefactorMirror (Liu et al., 2025) is a Java-only benchmark that checks whether LLM refactoring suggestions
align with human preferences. They observe generally low
alignment with human preferences, but do not verify the
proposal by having the LLM implement the refactoring.
Cordeiro et al. (2024) evaluate the ability of Starcoder to
decrease generic code complexity metrics on 30 Java tasks.
Rejiba et al. (2025) measure code complexity metrics when
tasking LLMs to refactor C++ and C code. The aider
team introduced a benchmark for Python code that evaluates function-level refactorings (Aider-AI, 2024). Finally,
Cassano et al. (2023) present instructions of varying degree
of detail to LLMs to refactor Python code on a functionlevel and measure whether generated code matches manually specified syntactic properties.

6. Conclusion
We introduced C ODE TASTE, a benchmark for evaluating
refactoring agents on real, multi-file changes mined from
open-source repositories. Our pipeline pairs each refactoring with executable environments and OpenGrep-based
static rules that verify the intended transformation beyond
test passing. Our benchmark features two complementary
tracks: an instructed track that measures diligence in the
execution of detailed refactorings, and an open track that
measures alignment with human refactoring choices under
underspecification.
Our results demonstrate that when provided with explicit
refactoring instructions, current agents achieve relatively
strong, yet unsaturated, performance. GPT-5.2 stands out
as the most capable model, achieving a compound alignment score A of 69.6%, leaving room for improvement
across the board. Alignment collapses when only a focus
area is provided. Propose-then-implement strategies and
oracle selection over multiple plans increase alignment by
up to 2.44-fold, yet remain far from human choices. These
findings highlight that autonomous judgment about what to
refactor remains a key bottleneck. We hope C ODE TASTE
serves as both a rigorous evaluation target and a preference
signal for training agents that can preserve long-term code
quality, and we see expanding the dataset, strengthening
rule and test coverage, and learning better proposal mechanisms as promising directions for future work.

A.2. Details of the execution environment agent
We use a polyglot container based on Ubuntu 24.04 with pre-configured toolchains and version managers for common
language ecosystems. The base image includes: (i) Python 3.8–3.11 managed via uv, (ii) Node.js 22.12.0 (default) managed via Node Version Manager (NVM), (iii) Go v1.23.4 and a Rust toolchain (cargo), (iv) OpenJDK, and (v) C/C++ via
GCC/Clang.
The build environment agent, S ONNET 4.5 in the C LAUDE C ODE harness, is restricted to $5 cost and 90 minutes time. To
ensure portability of the benchmark, we limit the size of the writable storage layer, after all setup is completed, to 5GB.
Further, we enforce a 15-minute time limit on running the test suite. This container is then committed.
In the runtime setup, we concretely remove the git commit history and remote origins to prevent leakage of golden commits.
