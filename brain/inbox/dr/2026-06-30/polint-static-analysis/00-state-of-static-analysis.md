## Conceptual Background
Static analysis tools inspect source code without executing it to find errors, enforce style, or verify security properties. The ecosystem includes several classes of tools.

| Term | Definition |
|---|---|
| SAST | Static Application Security Testing; tools that analyze source code for security vulnerabilities. |
| Linter | Tool that flags stylistic and programming errors in code. |
| Model Checker | Tool that verifies properties of a system by exploring its state space, often using mathematical solvers. |
| Metalinter | A tool or framework that aggregates or runs multiple linters and analyzers. |

## Findings
The registry categorizes static analysis tools primarily by programming language (e.g., C, C++, Java, JavaScript) and by function, including "Security/SAST", "Build tools", and "Metalinter" [S4]. This structure reflects a landscape where teams often select language-specific linters alongside general security scanners.

Within the C/C++ ecosystem, tools range from basic style checkers to deep analyzers. `cpplint` serves as an automated checker following Google's style guide, while `cppcheck` performs general static analysis [S4]. `clang-tidy` functions as a Clang-based linter with a limited ability to fix issues [S4]. At the high-assurance end, `ESBMC` operates as a context-bounded model checker based on satisfiability modulo theories [S4]. `Astrée`, a proprietary tool, automatically proves the absence of runtime errors and invalid concurrent behavior in C/C++ applications and is sound for floating-point computations [S4].
